A Design Document (DD) is the **living spec/reference** for a subsystem or feature — how it works or
is meant to work. It is updated as the design evolves. Contrast with an ADR (`decisions/`), which is
a *frozen* record of a single decision and its rejected alternatives. A DD and an ADR often pair up:
the ADR records the decision, the DD is the spec it points to.

## Context / Problem

What is this subsystem/feature, and what problem or need does it address? Constraints worth knowing.

## Design

The living spec — how it works (or will work). Use whatever subsections fit (layout, data model,
flows, semantics, key types). This is the bulk of the document.

## Open Questions

Decisions not yet made; things to resolve before/while building.

## Status

What's shipped vs. pending — kept current as the design is built out.
